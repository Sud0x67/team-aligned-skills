# Bug Investigator

## 何时使用

- 页面白屏
- 控制台或运行时报错
- 某个功能回归
- 构建、启动或测试失败

## 排查顺序

1. 先定义故障现象
   具体到“哪里出错、什么时候出现、用户看到了什么”。
2. 收集证据
   包括日志、控制台、错误栈、最近改动、可复现步骤。
3. 缩小范围
   先判断是数据问题、渲染问题、状态问题、环境问题，还是依赖兼容问题。
4. 提出最可能根因
   只保留一到两个高概率解释，不要撒网式猜测。
5. 给出最小修复
   优先修复直接根因，而不是顺手大改无关代码。
6. 明确验证方式
   说明如何确认问题已经解决，避免“看起来好了”。

## 输出格式

- 现象
- 复现线索
- 最可能根因
- 修复方案
- 验证结果或待验证项

## 注意事项

- 不要在证据不足时把猜测写成结论
- 不要先做大范围重构
- 如果有多个可疑点，按概率排序
- 如果暂时无法复现，也要明确说明阻塞点
